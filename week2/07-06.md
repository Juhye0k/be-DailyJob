# be-DailyJob

  ###### 1. 오늘 완료한 것
  - 기존 GET 방식의 회원가입 로직을 POST로 변경하였다.
  - HTTP 상태코드와 메서드를 enum으로 정의하여 타입 안정성과 재사용성을 높였다.
  - 회원가입 성공 시 302로 리다이렉트 하도록 설정했다.
  - 파서를 인터페이스로 구현하고 구현체들이 상속하는 구조를 설계하였다.
  - HttpRequest 필드에 byte[] body 필드를 추가하였다.

  - 아직 끝내지 못한 부분:
    - 파트 5,6,7 과제

  ###### 2. 오늘 가장 의미있었던 기술적 판단
  - 내가 내린 판단:
    - HTTP body를 파싱하기 위해 인터페이스를 구현하고 구현체를 분리하는 판단

  - 그렇게 판단한 이유:
    - Request는 byte 형태로 들어온다.
    - body는 key-value 뿐만 아니라 multipart/form-data, 이미지 등등 다앙한 형식이 들어오게 된다.
    - 요청 객체 설계 시점에서는 들어오는 객체의 타입을 알 수 없기 때문에 타입을 지정할 수 없다.
    - 타입을 지정해놓고 형변환 하는 것은 위험할 수 있다. 디코딩 후 인코딩 하는 과정에서 바이트 파일의 손상이 갈 수 있기 때문이다.
    - 따라서 파서가 컨텐츠 타입을 지원하는지 검사하고, 파서를 실행시킨다. 

  - 고려했지만 선택하지 않은 대안:
    - Object는 모든 객체의 부모 타입이기 때문에 고려를 했다. 그러나 Object로 받게 되면 명시적으로 타입 캐스팅을 해야한다. 업캐스팅 같은 경우는 컴파일러가 자동으로 캐스팅을 걸어주지만, 다운 캐스팅의 경우
      컴파일 오류가 발생할 수 있기 때문에 명시적으로 설정해야 한다.
    

  - 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
    - 파서는 인터페이스이기 때문에 핸들러는 구현체에 의존하지 않아 개방-폐쇠원칙에 부합한다.
    - 유지보수가 편리한 구조이다.

  ###### 3. 가장 막혔던 문제와 해결 과정
  - 문제 상황 또는 증상:
    - 파서 타입을 어떤 타입으로 받아야할지 고민이었다.

  - 처음 추측한 원인:
    - 와일드카드로 받으면 되지 않을까? 라고 생각했다.

  - 원인을 확인하기 위해 시도한 방법:
    - 와일드카드 방식을 고려하였다.

  - 실제 원인:
    - 이미지 파일을 받을 수 있는 Part 객체를 생성해야 했다.

  - 해결 방법:
    - Map<String, Part>를 통해 다양한 타입을 받을 수 있도록 하였다.
      

  ###### 4. 오늘 잘한 행동(개발과 관련된 것)
  - 어떤 행동이었는지:
    - 파서 인터페이스 구현

  - 그 접근이 도움이 된 이유:
    - 유지보수가 깔끔해졌다

  - 다음에도 반복하고 싶은 점:
    - 인터페이스를 통해 구현체에 의존하지 않는 확장가능한 코드 설계 방법에 대해서 고민해봐야겠다.

  ###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
  - 비효율적이었던 접근/놓치거나 부족했던 점:
    - body 파서를 구현할 때 아이디어는 있었으나, 자바 문법을 알지못해서 AI에게 구현을 부탁했다.

  - 그렇게 된 원인:
    - 자바 문법을 알지 못했다.
    - MIME 타입 바디에 따른 데이터 형식을 알지 못했다.

  - 다음에는 다르게 해볼 구체적인 행동:
    - 각 타입별 필요한 데이터 형식을 공식문서를 통해 파악한다.
    - 필요한 문법이 무엇인지 구글링을 해보거나, AI에게 힌트를 얻어 직접 구현해본다.
