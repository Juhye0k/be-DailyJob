
### 1. 오늘 완료한 것

* 오늘 실제로 끝낸 기능/수정/학습:

  * `Post`, `Like`, `Comment` 도메인의 기반 구조를 추가했다.
  * HikariCP 커넥션 풀과 `schema.sql`을 추가했다.
  * 기존 `Database` 클래스를 `UserRepository`로 변경해 회원 데이터 접근 책임을 명확히 했다.
  * `HandlerMapping`에 `/posts/{postId}`와 같은 경로 변수 매칭 기능을 추가했다.
  * 경로 변수 매칭에 대한 테스트를 작성했다.
  * `RequestContext`, `Authentication`, `SessionAuthenticator`를 도입했다.
  * HTTP 요청, 인증 정보, 경로 변수를 하나의 컨텍스트로 핸들러에 전달하도록 구조를 변경했다.
  * 게시글 상세 조회 기능을 구현했다.
  * 잘못된 `postId` 요청을 `400 Bad Request`로 처리했다.
  * 존재하지 않는 게시글 요청을 `404 Not Found` 페이지로 처리했다.

* (오늘의 목표 중에서) 아직 끝내지 못한 부분:

  * 댓글 등록과 좋아요 기능은 클래스와 도메인 구조만 일부 만들어진 상태다.
  * 댓글과 좋아요 기능의 라우팅 및 실제 동작은 완성하지 못했다.
  * 게시글 상세 조회 결과를 HTML 화면에 충분히 반영하는 작업이 남아 있다.

### 2. 오늘 가장 의미있었던 기술적 판단

* 내가 내린 판단:

  * 핸들러에 `HttpRequest`만 전달하던 구조를 변경하고, 요청·인증 정보·경로 변수를 함께 담는 `RequestContext`를 도입했다.

* 그렇게 판단한 이유:

  * 세션 쿠키를 해석하는 로직이 여러 컨트롤러에서 반복되고 있었다.
  * `/posts/{postId}`와 같은 동적 경로에서 추출한 값을 컨트롤러까지 전달할 공통 구조가 필요했다.
  * 컨트롤러가 요청 경로와 세션을 직접 해석하도록 두면 라우팅과 인증 책임이 컨트롤러에 섞이게 되기 때문이다.

* 고려했지만 선택하지 않은 대안:

  * 각 컨트롤러에서 `request.path()` 문자열을 직접 잘라 `postId`를 추출하는 방식이다.
  * 인증이 필요한 메서드마다 세션 쿠키를 조회하고 인증 여부를 반복해서 검사하는 방식이다.

* 이 판단이 코드 품질/테스트/유지보수에 미친 영향:

  * 인증 정보와 경로 변수 전달 책임이 `RequestHandler`와 `RequestContext`에 모였다.
  * 컨트롤러에서 반복되던 세션 처리와 경로 파싱 코드를 줄일 수 있었다.
  * 이후 다른 동적 경로나 인증이 필요한 기능에서도 동일한 구조를 재사용할 수 있게 되었다.
  * 다만 `Handler.handle()`의 파라미터 타입이 변경되면서 기존 컨트롤러와 테스트를 함께 수정해야 했고 변경 범위가 커졌다.

### 3. 가장 막혔던 문제와 해결 과정

* 문제 상황 또는 증상:

  * `/posts/{postId}`와 같은 동적 URL 처리와 로그인 여부 확인을 기존 요청 처리 구조에 자연스럽게 추가하기 어려웠다.

* 처음 추측한 원인:

  * `PageController`에서 요청 URL 문자열을 직접 잘라 사용하면 해결할 수 있다고 생각했다.

* 원인을 확인하기 위해 시도한 방법:

  * 기존 `HandlerMapping`이 HTTP 메서드와 경로를 어떤 방식으로 비교하는지 확인했다.
  * 인증 로직이 어느 클래스에서 반복되고 있는지 확인했다.
  * 라우터에서 생성된 정보가 핸들러와 컨트롤러까지 전달되는 구조인지 확인했다.

* 실제 원인:

  * 기존 라우팅 계층은 요청 경로와 등록 경로가 정확히 같은지만 비교하고 있었다.
  * 라우팅 계층에서 경로 변수를 추출할 수 없었다.
  * 핸들러에 전달되는 객체에 인증 정보와 경로 변수를 담을 공간이 없었다.

* 해결 방법:

  * `HandlerMapping`이 경로를 세그먼트 단위로 나누어 비교하도록 변경했다.
  * `{postId}`와 같은 세그먼트를 경로 변수로 인식하고 `Map<String, String>`에 저장하도록 구현했다.
  * 매칭 결과를 `Optional<Map<String, String>>`으로 반환하도록 변경했다.
  * `RequestContext`에 HTTP 요청, 인증 정보, 경로 변수를 담아 핸들러에 전달했다.

* 다음에 비슷한 문제가 생기면 먼저 확인할 것:

  * 컨트롤러에서 임시로 문자열을 파싱하기 전에, 필요한 정보가 요청 처리 파이프라인의 어느 단계에서 생성되어야 하는지 먼저 확인한다.
  * 공통 정보가 여러 컨트롤러에서 필요하다면 전달 구조를 먼저 설계한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)

* 어떤 행동이었는지:

  * 경로 변수 기능을 구현하면서 `HandlerMappingTest`를 함께 작성했다.
  * 정상 매칭뿐 아니라 HTTP 메서드 불일치, 세그먼트 개수 불일치, 여러 세그먼트가 있는 경로도 테스트했다.

* 그 접근이 도움이 된 이유:

  * 라우팅은 여러 기능에 영향을 주는 공통 인프라 코드이기 때문에 작은 오류도 전체 요청 처리에 영향을 줄 수 있다.
  * 경계 조건을 테스트로 고정해 이후 리팩토링 과정에서 발생할 수 있는 회귀를 줄였다.

* 다음에도 반복하고 싶은 점:

  * 공통 인프라 코드를 변경할 때는 정상 동작만 확인하지 않고 실패 조건과 경계 조건을 함께 테스트한다.
  * 라우팅, 인증, 요청 파싱처럼 영향 범위가 큰 코드는 구현과 테스트를 같은 작업 단위로 묶는다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

* 비효율적이었던 접근/놓치거나 부족했던 점:

  * 경로 변수, 인증 컨텍스트, 핸들러 시그니처, 게시글 상세 조회를 한 번에 변경해 작업 범위가 커졌다.
  * 여러 기반 구조를 동시에 수정하면서 각 변경의 완료 기준이 불명확해졌다.

* 그렇게 된 원인:

  * 게시글 상세 조회 기능을 구현하기 시작한 뒤, 이를 위해 필요한 라우팅과 인증 구조의 부족함이 한꺼번에 드러났다.
  * 기능을 시작하기 전에 필요한 기반 작업을 충분히 분리하지 못했다.

* 다음에는 다르게 해볼 구체적인 행동:

  * 작업을 다음과 같이 더 작은 단위로 나눈다.

    1. 경로 변수 매칭 구현 및 테스트
    2. `RequestContext` 도입 및 기존 핸들러 변경
    3. 인증 정보 전달 구조 검증
    4. 게시글 상세 조회 구현
    5. 상세 화면 데이터 반영
    6. 댓글 등록 구현
    7. 좋아요 기능 구현
  * 각 단계마다 테스트 통과 또는 브라우저 실행 확인과 같은 완료 기준을 하나씩 둔다.
