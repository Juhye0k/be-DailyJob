### 2. 오늘 가장 의미있었던 기술적 판단

* 내가 내린 판단:

  * HTTP body 파싱 책임을 인터페이스로 분리하고, Content-Type에 따라 적절한 파서 구현체가 처리하도록 설계했다.

* 그렇게 판단한 이유:

  * HTTP Request body는 byte 형태로 들어온다.
  * body에는 단순 key-value 데이터뿐 아니라 `multipart/form-data`, 이미지 파일 등 다양한 형식이 들어올 수 있다.
  * 요청 객체 설계 시점에서는 body의 구체적인 타입을 알 수 없기 때문에 하나의 타입으로 고정하기 어렵다.
  * 문자열로 디코딩한 뒤 다시 인코딩하면 이미지 같은 바이트 파일이 손상될 수 있다.
  * 따라서 원본 body는 `byte[]`로 유지하고, 각 파서가 자신이 지원하는 Content-Type인지 확인한 뒤 파싱하도록 설계했다.

* 고려했지만 선택하지 않은 대안:

  * `Object` 타입으로 body를 받는 방법을 고려했다.
  * 하지만 `Object`로 받으면 사용하는 쪽에서 명시적인 다운캐스팅이 필요하고, 잘못 캐스팅하면 런타임 오류가 발생할 수 있다.
  * 타입이 명확하지 않아 코드 가독성과 유지보수성이 떨어질 수 있다고 판단했다.

* 이 판단이 코드 품질/테스트/유지보수에 미친 영향:

  * 핸들러가 구체적인 파서 구현체에 직접 의존하지 않게 되었다.
  * 새로운 Content-Type을 지원해야 할 때 기존 코드를 크게 수정하지 않고 파서 구현체를 추가하면 된다.
  * 개방-폐쇄 원칙에 더 가까운 구조가 되었다.

### 3. 가장 막혔던 문제와 해결 과정

* 문제 상황 또는 증상:

  * HTTP body 파싱 결과를 어떤 타입으로 표현해야 할지 고민했다.

* 처음 추측한 원인:

  * 다양한 타입을 받을 수 있어야 하므로 와일드카드나 `Object`를 사용하면 되지 않을까 생각했다.

* 원인을 확인하기 위해 시도한 방법:

  * 와일드카드 방식과 `Object` 방식의 장단점을 비교했다.
  * body에 key-value 데이터뿐 아니라 파일 데이터도 들어올 수 있다는 점을 다시 확인했다.

* 실제 원인:

  * HTTP body는 Content-Type에 따라 구조가 달라진다.
  * 특히 이미지 파일 같은 바이너리 데이터는 단순 문자열이나 일반 객체로 다루면 손상될 수 있다.
  * 파일 데이터까지 표현하려면 별도의 `Part` 객체가 필요했다.

* 해결 방법:

  * 원본 body는 `byte[]`로 보관했다.
  * 파싱 결과는 `Map<String, Part>` 형태로 표현해 일반 필드와 파일 데이터를 함께 다룰 수 있도록 했다.
  * 각 body parser가 자신이 지원하는 Content-Type을 확인하고 파싱하도록 분리했다.

* 다음에 비슷한 문제가 생기면 먼저 확인할 것:

  * 데이터가 실제로 어떤 형식으로 들어오는지 먼저 확인한다.
  * Content-Type별 body 구조를 공식 문서나 HTTP 스펙 기준으로 확인한다.
  * 타입을 먼저 정하기보다, 데이터의 원본 형태와 사용 시점을 먼저 정리한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)

* 어떤 행동이었는지:

  * HTTP body parser를 인터페이스로 분리하고 구현체를 통해 확장 가능한 구조로 설계했다.

* 그 접근이 도움이 된 이유:

  * 핸들러가 구체적인 파서 구현체를 몰라도 되기 때문에 코드 결합도가 낮아졌다.
  * 새로운 body 형식이 추가되어도 기존 로직을 크게 바꾸지 않아도 된다.

* 다음에도 반복하고 싶은 점:

  * 기능을 바로 구현하기 전에 “이 책임이 어디에 있어야 하는지” 먼저 고민한다.
  * 구현체에 직접 의존하기보다 인터페이스로 역할을 분리할 수 있는지 확인한다.

---

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

* 비효율적이었던 접근/놓치거나 부족했던 점:

  * body parser 구현 아이디어는 있었지만, 자바 문법과 MIME 타입별 body 구조를 충분히 이해하지 못해 AI에게 구현을 많이 의존했다.

* 그렇게 된 원인:

  * 자바의 타입 시스템, 제네릭, 다운캐스팅에 대한 이해가 부족했다.
  * `multipart/form-data`와 같은 MIME 타입의 실제 데이터 구조를 정확히 알지 못했다.

* 다음에는 다르게 해볼 구체적인 행동:

  * 먼저 Content-Type별 body 구조를 공식 문서나 예제 요청으로 확인한다.
  * 필요한 자바 문법을 직접 찾아보고 작은 예제 코드로 먼저 실험한다.
  * AI에게 바로 구현을 맡기기보다, 먼저 직접 설계와 초안을 작성한 뒤 막히는 부분만 질문한다.
