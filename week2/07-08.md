> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것

* 오늘 실제로 끝낸 기능/수정/학습:

  * `BusinessException`, `ErrorCode`, `GlobalExceptionHandler`를 활용한 전역 예외 처리 구조를 구현했다.
  * 회원가입과 로그인 실패 응답을 동일한 방식으로 처리하도록 통일했다.
  * H2 데이터베이스 설정을 추가했다.
  * 회원 정보를 인메모리 `Map`이 아니라 데이터베이스에 저장하고 조회하도록 변경했다.
  * 아이디와 닉네임 중복 검사를 구현했다.
  * 로그인 실패 원인을 아이디 없음과 비밀번호 불일치로 분리했다.

* (오늘의 목표 중에서) 아직 끝내지 못한 부분:

  * 데이터베이스 초기화 코드는 추가했지만 `schema.sql`은 작성하지 못했다.
  * 전역 예외 처리 흐름에 대한 테스트가 부족했다.
  * 데이터베이스 설정부터 초기 스키마 생성까지 하나의 실행 가능한 작업 단위로 완성하지 못했다.

### 2. 오늘 가장 의미있었던 기술적 판단

* 내가 내린 판단:

  * 컨트롤러가 실패 응답을 직접 생성하지 않고, 비즈니스 실패를 예외로 표현한 뒤 전역 예외 처리기가 HTTP 응답으로 변환하도록 설계했다.

* 그렇게 판단한 이유:

  * 회원가입 중복, 로그인 실패, 잘못된 요청과 같은 실패 처리가 여러 컨트롤러에 흩어지면 동일한 응답 생성 코드가 반복되기 때문이다.
  * 컨트롤러는 비즈니스 흐름에 집중하고, 실패 응답 변환은 한 곳에서 관리하는 것이 책임 분리에 더 적합하다고 판단했다.

* 고려했지만 선택하지 않은 대안:

  * 각 컨트롤러에서 `HttpResponse.alert()` 또는 `redirect()`를 직접 반환하는 방식이다.

* 이 판단이 코드 품질/테스트/유지보수에 미친 영향:

  * 실패 응답 형식을 한 곳에서 일관되게 관리할 수 있게 되었다.
  * 컨트롤러의 책임이 줄어들어 단일 책임 원칙 측면에서 구조가 개선되었다.
  * 새로운 예외가 추가되더라도 전역 예외 처리기를 중심으로 확장할 수 있게 되었다.
  * 다만 `ErrorCode`가 메시지와 리다이렉트 경로를 함께 가지면서 예외 코드와 화면 흐름이 강하게 결합되는 문제가 생겼다.

### 3. 가장 막혔던 문제와 해결 과정

* 문제 상황 또는 증상:

  * 전역 예외 처리기에서 예외를 처리했지만, 생성한 응답이 실제 클라이언트에게 전달되지 않았다.

* 처음 추측한 원인:

  * 예외 타입이 올바르게 잡히지 않거나 `ErrorCode` 매핑이 부족한 문제라고 생각했다.

* 원인을 확인하기 위해 시도한 방법:

  * `RequestHandler`에서 예외가 발생한 이후 `GlobalExceptionHandler`를 거쳐 `HttpResponseWriter.write()`까지 실행되는 흐름을 확인했다.
  * 예외 처리기에서 응답 객체를 생성한 뒤 실제 반환하거나 출력하는지 확인했다.

* 실제 원인:

  * `GlobalExceptionHandler.handle()`이 `HttpResponse`를 반환하지 않고 내부에서 응답 객체만 생성한 뒤 사용하지 않고 있었다.
  * 응답 객체는 생성되었지만 출력 스트림에 기록되지 않았다.

* 해결 방법:

  * `GlobalExceptionHandler.handle()`이 `HttpResponse`를 반환하도록 변경했다.
  * `RequestHandler`가 정상 처리와 예외 처리 모두에서 최종적으로 한 번만 `HttpResponseWriter.write()`를 호출하도록 수정했다.

* 다음에 비슷한 문제가 생기면 먼저 확인할 것:

  * 응답 객체가 생성되었는지만 보지 않고, 실제 출력 스트림에 `write`까지 수행되었는지 먼저 확인한다.
  * 요청 처리 흐름에서 최종 응답 작성 책임이 어느 객체에 있는지 먼저 확인한다.

### 4. 오늘 잘한 행동(개발과 관련된 것)

* 어떤 행동이었는지:

  * 로그인 실패를 단순한 `boolean` 값으로 처리하지 않고, 아이디 없음과 비밀번호 불일치를 서로 다른 실패 원인으로 분리했다.

* 그 접근이 도움이 된 이유:

  * 코드만 보더라도 실패 원인을 명확하게 알 수 있게 되었다.
  * 사용자에게 상황에 맞는 메시지를 제공할 수 있게 되었다.
  * 이후 로그인 실패 정책이나 예외 종류를 확장하기 쉬워졌다.

* 다음에도 반복하고 싶은 점:

  * 여러 의미를 하나의 `boolean` 값으로 뭉개지 않고, 도메인에서 구분해야 하는 실패 원인을 먼저 나눈다.
  * 실패 결과도 정상 결과만큼 구체적인 타입과 이름으로 표현한다.

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)

* 비효율적이었던 접근/놓치거나 부족했던 점:

  * 데이터베이스 연결 코드와 초기 스키마 파일을 하나의 작업 단위로 완성하지 못했다.
  * 코드 작성에 집중하면서 실제 실행에 필요한 리소스 파일을 놓쳤다.

* 그렇게 된 원인:

  * 데이터베이스 기능을 설정 코드 작성 정도로만 생각하고, 의존성·설정·스키마·실행 확인까지 하나의 기능으로 보지 못했다.

* 다음에는 다르게 해볼 구체적인 행동:

  * 데이터베이스 기능을 추가할 때 다음 항목을 하나의 체크리스트로 확인한다.

    * 의존성 추가
    * 데이터베이스 설정
    * `schema.sql` 작성
    * 애플리케이션 실행
    * 테이블 생성 확인
    * 데이터 저장 및 조회 확인
