###### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단: Request, Response 분리와 Reader,Writer의 분리
- 그렇게 판단한 이유: 기존에는 파싱 시 변수를 정의하여 각각 관리하는 방식을 하였다. 그러나 HTTP request line과 헤더, 본문 등 자료가 많기 때문에 처리 메서드 안에서 파싱하면 코드가 매우 길어질 것이라고 판단했다. 따라서 요청, 응답별로 객체를 만들어 필드를 내부에서 관리하는
  방식을 생각했다. 객체와 파서를 분리한 이유는 단일 책임 원칙을 지키기 위해 분리하였다. 객체 자체에 해당하는 것과 또 객체를 파싱하는 부분을 분리해서 관리하는 것이 유지보수가 용이하다고 판단했다.
- 고려했지만 선택하지 않은 대안: 객체 내부에 파서 메서드를 추가할까 생각했지만 코드가 길어질 것이라고 생각했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향: 관심사 필드를 한 객체에 모아서 관리하기 용이하다

 ### 1. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트

  원본 프롬프트:

                                                                                                        
  > 실제 소켓을 생성하지 말고 mock으로 테스트하는 방법을 생각 중인데, 가장 좋은 테스트 방법을 알려줘.                                                                                  

  선택한 이유:

  이 프롬프트는 단순히 “테스트 코드 작성해줘”라고 요청하지 않고, 학습 목표, 기능 요구사항, 테스트 조건까지 함께 제공했다.
  특히 “실제 소켓을 생성하지 않고 테스트하고 싶다”는 제약을 명확히 제시했기 때문에, AI가 Socket을 mock하는 방식보다 StringReader와 BufferedReader를 활용해 HttpRequestReader를 단위 테
  스트하는 방향을 제안할 수 있었다.


  ### 2. 원본 프롬프트 구조 분석

  - 문제 상황:
    HTTP GET 요청으로 회원가입 데이터를 전달받고, query parameter를 파싱해야 하는 상황이다. 또한 이 기능을 테스트해야 한다.

  - 원하는 결과를 어떻게 요구했는가:
    JUnit과 AssertJ를 활용한 단위 테스트 작성 방법을 요청했다.

  - 필요한 맥락이 포함되어 있는가:
    학습 목표, 기능 요구사항, HTTP Request 예시, query parameter 추출 힌트가 포함되어 있어 충분한 맥락이 있었다.

  - 판단 기준이나 제약 조건이 있는가:
    “실제 소켓을 생성하지 말 것”이라는 제약이 있었다. 이 제약 덕분에 테스트 대상이 RequestHandler.run()이 아니라 HttpRequestReader로 좁혀졌다.

  - 출력 방식이 명확한가:
    테스트 코드를 원한다는 점은 명확했지만, “어떤 클래스의 테스트인지”, “성공 가능한 테스트만 원하는지”, “실패 테스트도 포함할지”는 처음에는 완전히 명확하지 않았다.

  - 나의 가설, 우려, 의도가 드러나는가:
    실제 소켓을 생성하지 않고 mock으로 테스트하려는 고민이 드러났다. 즉, 네트워크 I/O를 직접 테스트하는 것이 부담스럽고, 더 좋은 단위 테스트 방법을 찾고 싶다는 의도가 드러났다.


  ### 3. AI에게 프롬프트를 개선 요청하기

  - 프롬프트 개선안

  현재 Java로 간단한 WAS를 구현하고 있습니다.

  학습 목표는 HTTP GET 요청을 이해하고, GET query parameter로 전달된 회원가입 데이터를 서버에서 파싱해 model.User 객체에 저장하는 것입니다.

  현재 요청 파싱 책임은 HttpRequestReader가 가지고 있고, BufferedReader를 입력으로 받아 HttpRequest를 반환합니다. 실제 Socket을 생성하거나 RequestHandler.run() 전체를 테스트하기보다
  는, 요청 파싱 로직을 단위 테스트하고 싶습니다.

  다음 내용을 만족하는 JUnit5 + AssertJ 테스트 코드를 작성해주세요.

  조건:
  - 실제 Socket은 사용하지 않는다.
  - StringReader와 BufferedReader를 사용해 HTTP 요청 원문을 테스트한다.
  - 테스트 대상은 HttpRequestReader.parseHttpRequest()로 한다.
  - 현재 구현에서 성공 가능한 테스트만 작성한다.
  - URL decoding은 아직 구현하지 않았으므로 디코딩 테스트는 포함하지 않는다.

  검증하고 싶은 항목:
  - GET 요청 라인에서 method, target, version을 파싱한다.
  - HTTP headers를 파싱한다.
  - query parameter를 파싱한다.
  - 헤더 값에 콜론(:)이 포함되어도 정상 파싱한다.

  테스트 파일은 src/test/java/webserver/HttpRequestReaderTest.java 위치에 작성하는 것을 기준으로 예시를 보여주세요.


  ### 4. AI가 개선한 프롬프트 분석

  - 원본 프롬프트에 비해 맥락이 더 명확해졌다.
    기존에는 “실제 소켓을 생성하지 않고 mock으로 하는 방법” 정도만 제시했지만, 개선안에서는 현재 구조가 HttpRequestReader -> HttpRequest 흐름이라는 점을 명시했다.

  - 목표가 더 구체화되었다.
    단순히 테스트 코드를 작성하는 것이 아니라, HttpRequestReader.parseHttpRequest()를 테스트 대상으로 지정했다.

  - 제약 조건이 추가되었다.
    실제 Socket을 사용하지 않고, StringReader와 BufferedReader를 사용해야 한다는 점을 명확히 했다.

  - 성공 가능한 테스트만 작성하도록 범위를 제한했다.
    URL decoding은 아직 구현하지 않았으므로 디코딩 테스트를 제외한다고 명시해, 테스트 작성 과정에서 원본 비즈니스 로직을 수정하지 않도록 했다.

  - 출력 형식이 명확해졌다.
    테스트 파일 위치와 테스트해야 할 항목을 구체적으로 제시해, AI가 바로 적용 가능한 테스트 코드를 작성하기 쉬워졌다.

  - 답변 품질이 좋아질 것으로 예상되는 이유:
    테스트 대상, 테스트 방식, 제외할 범위, 검증 항목이 모두 명확하기 때문에 AI가 불필요하게 Socket mock을 제안하거나 구현 코드를 수정하는 방향으로 벗어날 가능성이 줄어든다.
