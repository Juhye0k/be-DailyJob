# be-DailyJob

  ###### 1. 오늘 완료한 것
  - 회원가입 페이지에서 GET 메서드를 통해 회원가입 요청을 보낼 수 있도록 구현했다.
  - 회원가입 요청(`/user/register?...`)을 받아 사용자 정보를 파싱하고 `Database`에 저장하는 흐름을 만들었다.
  - 회원가입 완료 후 `302 Redirect`를 통해 메인 페이지(`/`)로 이동하도록 처리했다.

  - 아직 끝내지 못한 부분:
    - 회원가입 로직에 대한 테스트 코드 작성

  ###### 2. 오늘 가장 의미있었던 기술적 판단
  - 내가 내린 판단:
    - 회원가입 완료 후 서버가 HTML을 직접 응답하지 않고, 리다이렉트를 통해 메인 페이지로 이동시키도록 했다.

  - 그렇게 판단한 이유:
    - 회원가입 요청은 페이지를 조회하는 요청이 아니라 사용자 데이터를 처리하는 요청이다.
    - 따라서 `/user/register`는 화면을 렌더링하는 책임보다 회원 데이터를 저장하는 책임에 집중하는 것이 맞다고 판단했다.
    - 데이터 처리가 끝난 후에는 브라우저에게 다시 `/`로 이동하라고 알려주는 방식이 역할 분리에 더 적합했다.

  - 고려했지만 선택하지 않은 대안:
    - 회원가입 처리 요청 안에서 바로 메인 페이지 HTML을 읽어 응답하는 방식은 선택하지 않았다.
    - 이 방식은 하나의 요청이 데이터 처리와 화면 렌더링을 동시에 담당하게 되어 책임이 섞인다고 판단했다.

  - 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
    - `response302Header`를 정의해 리다이렉트 응답을 재사용할 수 있게 되었다.
    - 이후 로그인, 게시글 작성 등 처리 후 특정 페이지로 이동해야 하는 기능에서도 같은 응답 로직을 활용할 수 있다.

  ###### 3. 가장 막혔던 문제와 해결 과정
  - 문제 상황 또는 증상:
    - `/user/register` 요청을 기존 정적 파일 요청과 같은 방식으로 처리하려고 하면서 문제가 발생했다.

  - 처음 추측한 원인:
    - 기존에는 URL에 확장자가 있는지 확인하거나 `split("\\.")`을 사용해 정적 파일 경로를 구분했다.
    - 하지만 `/user/register`는 실제 HTML/CSS 파일을 요청하는 경로가 아니라 회원가입 처리를 위한 API 성격의 요청이었다.

  - 원인을 확인하기 위해 시도한 방법:
    - 요청 URL의 형태를 비교했다.
    - `/index.html`, `/main.css`, `/img/like.svg` 같은 정적 파일 요청과 `/user/register?...` 같은 회원가입 처리 요청의 목적이 다르다는 점을 확인했다.

  - 실제 원인:
    - 정적 파일 요청과 동적 요청을 구분하지 않고 같은 파일 탐색 흐름에 태우려고 한 것이 원인이었다.
    - `/user/register`는 파일을 찾는 요청이 아니라 파라미터를 읽고 데이터를 저장해야 하는 요청이었다.

  - 해결 방법:
    - `/user/register`로 시작하는 요청은 먼저 동적 요청으로 처리하도록 분기했다.
    - 회원가입 요청이면 query string을 파싱해 `User` 객체를 만들고 `Database`에 저장했다.
    - 처리가 끝나면 `302 Redirect` 응답을 보내고 정적 파일 처리 로직으로 넘어가지 않도록 했다.

  - 다음에 비슷한 문제가 생기면 먼저 확인할 것:
    - URL 문자열 자체보다 HTTP 요청의 목적을 먼저 파악한다.

  ###### 4. 오늘 잘한 행동(개발과 관련된 것)
  - 어떤 행동이었는지:
    - 리다이렉트 응답을 `response302Header` 메서드로 분리했다.

  - 그 접근이 도움이 된 이유:
    - 회원가입 완료 후 브라우저를 다른 페이지로 이동시키기 위해 `302 Found`와 `Location` 헤더가 필요했다.
    - 이 응답을 매번 직접 작성하면 중복이 생기고 실수할 가능성이 커진다.
    - 별도 메서드로 분리하면서 `RequestHandler`의 흐름이 조금 더 명확해졌다.

  - 다음에도 반복하고 싶은 점:
    - `200 OK`, `302 Found`처럼 응답 형태가 달라질 때마다 응답 생성 책임을 메서드로 분리한다.
    - 요청 처리 로직과 응답 작성 로직이 한곳에 과도하게 섞이지 않도록 신경 쓴다.

  ###### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
  - 비효율적이었던 접근/놓치거나 부족했던 점:
    - 처음에는 모든 URL을 정적 파일 경로처럼 처리하려고 했다.
    - 그 결과 `/user/register`, `/index.html`, `/registration`처럼 성격이 다른 요청을 한 흐름에서 처리하려고 하면서 코드가 복잡해졌다.

  - 그렇게 된 원인:
    - HTTP 요청을 “URL 문자열” 중심으로만 바라보고, 요청의 목적을 충분히 구분하지 못했다.
    - 정적 요청과 동적 요청을 나누는 기준을 처음부터 명확히 세우지 못했다.

  - 다음에는 다르게 해볼 구체적인 행동:
    - 요청을 처리하기 전에 먼저 다음 기준으로 분류한다.
      - 정적 파일 요청인지
      - 페이지 라우팅 요청인지
      - 데이터 처리 요청인지
    - `RequestHandler` 안에 모든 로직을 넣기보다, 회원가입 같은 비즈니스 로직은 `UserService` 같은 별도 클래스로 분리하는 방향을 고려한다.
    - 회원 저장 로직을 분리한 뒤 JUnit과 AssertJ로 단위 테스트를 작성한다.
